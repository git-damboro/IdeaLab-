# 参考文献正文标注位置建议

## 标注原则

1. 正文引用编号应按照参考文献列表顺序依次出现，即先出现 `[1]`，再出现 `[2]`，以此类推。
2. 多篇文献同时支撑同一句话时，不使用 `[3-5]` 这种连续写法，应写为 `[3][4][5]`。
3. 每一条参考文献至少在正文中出现一次。
4. 尽量将参考文献放在与其主题最接近的位置，避免为了凑引用而放在无关段落。
5. 你当前参考文献中 `[4]` 与 `[14]` 内容高度重复，正式定稿时建议删除其中一条；如果学校要求保持 15 条，则可以分别在不同位置引用。

## 按正文顺序插入后的参考文献标注示例

下面按照你的正文段落顺序，给出可以直接替换或插入的示例句。

### 1.1.1 研究背景

```text
随着大语言模型和向量检索技术的发展，文献检索系统开始从“字符串匹配”向“语义理解与知识组织”转变[1]。通过将文献摘要、标题等内容编码为高维向量，可以在一定程度上弥补关键词检索在语义层面的不足；再结合生成式模型进行问答和推荐解释，可以进一步提升检索系统的可理解性和交互性。这种“检索+生成”的组合模式，为构建更智能的学术信息服务平台提供了新的实现路径[2]。
```

### 1.1.2 研究意义

```text
在此基础上，系统进一步引入大语言模型，为用户提供推荐理由生成与智能问答服务[3]，从而使文献检索过程从单纯的信息查找扩展为结果理解与知识辅助过程。
```

```text
本系统通过将关键词检索、语义向量检索、推荐摘要和智能问答结合起来，帮助用户更高效地完成学术资料的检索、筛选、阅读与整理[4]，在一定程度上缓解海量文献持续增长带来的认知压力和时间成本。
```

```text
本课题验证了轻量级 RAG 架构在低资源环境中的可实现性与实用性[5]，具有较强的工程应用价值。
```

```text
本课题通过将用户检索、收藏、笔记、推荐摘要、智能问答以及后台管理审核等功能整合到统一平台中，形成了较完整的业务闭环[6]，具有一定的实践探索价值。
```

### 2.1.2 FastAPI框架

```text
本系统后端采用 FastAPI 框架实现[7]。FastAPI 是基于 Python 的现代化 Web 开发框架，主要用于构建高性能 API 服务。
```

### 2.1.5 RAG技术

```text
与单纯依赖模型参数记忆的生成方式相比，RAG 能够有效缓解大模型在专业领域中的知识不足和“幻觉”问题，使生成结果更加贴合真实数据来源[8]。
```

### 2.1.4 Milvus

```text
随着人工智能和自然语言处理技术的发展，越来越多的文本、图像和语音数据会被表示为向量形式[9]，而传统数据库在高维向量检索方面效率较低。
```

### 2.1.5 RAG技术

```text
在智能文献检索系统中，RAG 技术能够根据用户提出的研究问题，先检索相关文献，再结合文献标题、摘要等信息生成回答或推荐说明[10]，从而实现“先检索、后生成”的智能服务模式。
```

```text
这不仅提升了系统的问答质量，也增强了用户对检索结果的理解能力，因此 RAG 技术已成为当前智能检索与知识问答系统中的重要实现方式[11]。
```

### 3.3.1 用户端功能需求

```text
该需求结构说明，文献检索系统的核心价值不仅在于返回结果，更在于支持用户从结果中提取有效信息并形成个人知识积累[12]。
```

### 4.1 系统总体架构设计

```text
后端通过模块化方式将用户认证、检索服务、问答服务、任务处理和权限控制等功能进行拆分，不同模块之间职责明确，便于后续接入新的检索算法、嵌入模型或大语言模型服务[13]。
```

### 5.1.3 收藏、笔记与智能问答

```text
当用户完成文献检索后，可进一步针对当前研究主题提出问题，系统会从已检索到的文献中提取关键信息，构造问答上下文，并调用大语言模型生成简洁回答[14]。
```

### 7.2 展望

```text
第三，深度优化 RAG 检索增强生成流程，精细化调整向量召回策略、上下文片段筛选逻辑与 prompt 工程设计，强化引用溯源标注，保障智能问答内容精准稳定、有据可依，降低模型幻觉问题[15]。
```
