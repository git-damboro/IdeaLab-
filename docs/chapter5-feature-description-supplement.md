# 第5章系统实现功能描述补充建议

## 一、总体判断

你当前第 5 章已经覆盖了用户端和管理端的大部分核心功能，包括登录注册、文献检索、收藏、笔记、智能问答、仪表盘、论文管理、任务中心、角色权限和审计管理等内容。整体结构是合理的，不需要大幅重写章节框架。

不过，从系统实际功能和论文完整性看，还可以补充以下几类内容：

| 补充方向 | 是否建议加入 | 原因 |
|---|---|---|
| 检索历史与查询扩展 | 建议加入 | 这是用户端检索体验的重要组成部分，当前正文提到较少 |
| 推荐摘要缓存机制 | 建议加入 | 能体现系统工程优化和大模型调用成本控制 |
| 论文详情与用户笔记联动 | 建议加入 | 可增强用户端功能闭环描述 |

| 用户端状态管理 | 可选加入 | 如果篇幅允许，可简单说明 Pinia/前端状态维护 |
| 后端模块化接口 | 可选加入 | 可放在 5.3 后端实现小节中，但如果论文第 4 章已写架构，可不展开 |

建议不是新增很多小节，而是在现有小节中补充若干段落。若需要新增小节，推荐增加一个 `5.3 后端接口与任务调度实现`，用于承接后端实现细节。

## 补充1：检索历史与查询扩展功能

| 插入位置 | 建议插入在 `5.1.2 文献检索与结果展示` 第一段之后 |
|---|---|

可插入正文如下：

```text
为了提高检索入口的易用性，系统在用户每次提交检索请求后，会将检索关键词、用户编号和检索时间写入搜索历史集合中，并在前端页面按时间倒序展示最近的检索记录。用户可以通过历史记录快速回顾先前的检索主题，减少重复输入成本。同时，后端在接收到用户问题后，并不直接使用原始文本进行单一路径查询，而是先对查询内容进行扩展处理，包括中文关键词提取、关键词组合以及部分关键词英文翻译。通过构造多个查询变体，系统能够同时兼顾中文自然语言表达和英文论文数据之间的差异，提高文献召回的完整性。

该设计使系统更适合真实学术检索场景。用户输入的问题往往具有口语化、概括性强的特点，而论文元数据中标题、摘要和关键词可能以英文或专业术语形式存在。查询扩展机制能够在不增加用户操作复杂度的前提下增强检索覆盖范围；搜索历史则为用户持续探索同一研究主题提供便利，使检索过程从一次性查询扩展为可追溯、可延续的研究活动。
```

## 补充2：推荐摘要缓存与后台生成机制

| 插入位置 | 建议插入在 `5.1.3 收藏、笔记与智能问答` 中“智能问答模块”之前 |
|---|---|

可插入正文如下：

```text
推荐摘要功能是连接检索结果与智能问答的重要中间环节。系统在返回论文列表时，会优先为当前页排序靠前的论文生成推荐原因，说明该论文与用户查询主题之间的关联。为了避免同一查询和同一论文反复调用大语言模型，后端设计了摘要缓存机制，将用户查询、论文编号和生成结果组合成缓存键，并将推荐摘要保存至 MongoDB 中。当用户再次查看相同检索结果时，系统优先读取缓存内容，从而降低模型调用次数，提高页面响应速度。

对于当前页以外的论文，系统采用后台任务方式逐步补充推荐摘要。该设计避免了首屏加载时一次性生成大量文本造成响应延迟，也降低了大模型接口调用失败对整体检索流程的影响。通过“当前页优先生成、其余结果后台补充、重复内容缓存复用”的策略，系统在用户体验、响应效率和模型调用成本之间取得了较好的平衡。
```

| 插入位置 | 建议插入在 `5.1.3 收藏、笔记与智能问答` 的笔记功能段落之后 |
|---|---|

可插入正文如下：

```text
在论文阅读与管理过程中，系统将笔记功能与具体论文记录进行关联。用户在查看某篇论文时，可以围绕该论文添加阅读笔记，记录研究方法、创新点、实验结论和个人理解等内容。后端通过用户编号和论文编号共同确定笔记归属，保证不同用户之间的数据相互隔离，同时也避免同一用户对同一论文产生重复笔记记录。用户再次打开该论文时，系统能够加载已有笔记并支持继续编辑。

该功能使系统不仅具备文献检索能力，也具备一定的知识管理能力。相比单纯收藏论文，笔记功能能够帮助用户沉淀阅读过程中的判断和思考，使系统更贴近毕业论文写作、课题调研和文献综述整理等实际使用场景。配合 Markdown 导出能力，用户还可以将系统中的阅读记录迁移到论文草稿或个人知识库中，提高文献利用效率。
```

## 补充4：Milvus 异常降级与检索容错

| 插入位置 | 建议插入在 `5.1.2 文献检索与结果展示` 末尾，或新增 `5.3.2 推荐摘要缓存与降级机制` |
|---|---|

可插入正文如下：

```text
考虑到向量数据库在本地部署环境中可能出现服务未启动、连接超时或资源不足等情况，系统在检索模块中加入了降级处理机制。后端启动时会尝试连接 Milvus，如果连接失败，则关闭向量检索通道，仅保留 BM25 关键词检索能力。这样即使向量数据库暂时不可用，用户仍然可以通过关键词方式获得基本检索结果，避免整个检索功能完全失效。

该降级机制提高了系统在低资源开发环境下的可用性。对于毕业设计和本地演示场景而言，MongoDB、Milvus、大语言模型接口等外部依赖较多，任何一个组件异常都可能影响系统运行。通过对 Milvus 连接状态进行检测，并在异常情况下自动切换为关键词检索，系统具备了一定的容错能力。虽然纯关键词检索在语义理解方面弱于混合检索，但能够保证系统核心功能可持续使用，为后续优化前端状态提示和检索策略说明奠定基础。
```

| 插入位置 | 建议插入在 `5.2.2 导入任务与日志跟踪` 的后半部分，或放入 `5.3.3 模块化接口与后台任务调度` |

```text
任务调度模块采用后台线程轮询方式实现。系统启动后，JobRunner 会定期扫描 `ingest_jobs` 集合中状态为 `queued` 的任务，并将其状态更新为 `running` 后执行具体任务逻辑。对于导入任务，系统会读取任务参数中的文件路径，按照文件类型选择对应解析方式，将论文元数据写入 MongoDB，并在需要时生成向量写入 Milvus。任务执行过程中，系统持续更新进度百分比和日志内容，使管理员能够在前端页面观察任务执行状态。
```
